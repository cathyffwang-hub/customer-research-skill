# 客户调研大纲生成 Skill

当用户说"我要调研 xx 客户，这是我的调研安排，帮我生成调研大纲"或类似表述时，按本 skill 定义的流程执行。

**运行前置检查（每次启动 skill 必须第一步执行）**：

```bash
node /Users/cathy/.workbuddy/skills/客户调研大纲生成/scripts/update-check.js
```

该脚本会自动 `git pull` 拉取最新 skill 代码，无需手动更新。

## 🧭 执行流程（严格按顺序）

### Step 1 · 信息收集（用结构化选项卡让用户"选"而不是"打字"）

**核心原则**：能让用户点选的一律用选项卡（`ask_followup_question` 工具），只有无法枚举的字段才让用户打字。

#### 1.1 信息清单（v1.7）

**核心原则**：
- **能枚举的字段 → 一律选项卡**（`ask_followup_question`），让用户点选
- **不能枚举的字段 → 纯文本**（客户名、日期、署名、调研人员、调研对象、材料名、场次排期明细），留空白让用户自己写
- 不硬塞预设值（如"cathyfwang/elgong"、"今天/明天"这种），避免误导

| # | 字段 | 收集方式 | 选项 / 示例 |
|---|------|---------|----------|
| 1 | **客户名称** | 📝 文本 | 首轮消息提取 or 追问，如"广联达" |
| 2 | **产品版本** | 🔘 选项卡 | ① 腾讯文档 ② 企微文档 |
| 3 | **调研署名（by-@）** | 📝 文本 | 用户自己写，如"cathyfwang"、"elgong" |
| 4 | **客户使用基础** | 🔘 选项卡 | ① A · 用得不好（场景挖掘型） ② B · 用得好（最佳实践型）|
| 5 | **调研日期** | 📝 文本 | 只要**年月日**，如 `2026-04-28` 或 `2026-04-28 至 2026-04-30`。具体"10:00-10:30"时段放**场次排期**里每场的"时间"列 |
| 6 | **调研地点** | 🔘 选项卡 | ① 现场办公室 ② 线上腾讯会议 ③ 线上企微会议 ④ 混合（选后用户自己补充说明）|
| 7 | **调研人员（我方）** | 📝 文本 | 用户自己写名单，多人用顿号分隔 |

| 9 | **调研背景** | ☑️ 选项卡（多选 + 自己写）| 见下方【调研背景候选场景】|
| 10 | **调研目的** | ☑️ 选项卡（多选 + 自己写）| 见下方【调研目的候选维度】|
| 11 | **场次排期明细** | 📝 文本/表格/图片 | 用户粘贴原始排期、截图或逐条描述，AI 自行解析（序号/时间/部门/对象/智能表场景）|
| 12 | **归档位置** | 🔘 选项卡 | ① 归档到「客户调研｜AI生成」/ 客户名子夹（**默认推荐**） ② 不归档，留在首页 ③ 归档到其他空间（选后追问空间名 + 文件夹名）|

**B 模板额外字段**（仅使用基础 = B 时收集）：

| # | 字段 | 收集方式 | 说明 |
|---|------|---------|------|
| 13 | **调研对象**（被调研人）| 📝 文本 | 如"市场部 · 张三" |
| 14 | **调研材料** | 📝 文本 | 如"2025Q1 电营收集表"，无则填"无" |

#### 1.1.1 背景/目的选项表（v1.4 新增）

**【调研背景】候选场景**（单选/多选，支持"自己写"）：
- ① 客户已推广使用一段时间，想挖掘深度场景
- ② 新签客户，想做场景诊断和铺开规划
- ③ 配合客户内部大赛/评选，收集优秀案例
- ④ 客户提了具体痛点，本次专项跟进
- ⑤ 续约/增购前置摸底调研
- ⑥ 自己写（选后再追问文本）

**【调研目的】候选维度**（多选，支持"自己写"）：
- ① 场景深度挖掘（了解各业务线使用场景）
- ② 需求收集（功能优化/未满足痛点）
- ③ 最佳实践沉淀（对内模板库 + 对外品牌推广案例）
- ④ 业务流程梳理（使用前后对比，量化提效）
- ⑤ 用户画像摸底（了解角色/部门/岗位）
- ⑥ 自己写（选后再追问文本）

> AI 根据用户选的标签，**自动拼成一段自然语句**填入"调研背景"和"调研目的"字段（不要直接把选项原文堆进去）。示例：
> - 用户选：背景③ + 目的①②③④
> - 背景段自动生成："「三七互娱」内部正在开展智能表格评选大赛，借此契机收集各部门的优秀使用案例。"
> - 目的段自动生成 4 条子目的列表。

#### 1.2 提问流程（v1.7：选项卡归选项卡，文本归文本）

**Step A · 信息盘点清单**：
AI 先用一张 Markdown 表格列出所有需要收集的信息清单，标明用户**首轮消息已提供**的项（✅），告诉用户"先点两批选项卡，再统一补几个文本信息"。

**Step B · 第一批选项卡**（可枚举的基础 4 项，`ask_followup_question` 一次 4 题）：
① 产品版本 / ② 使用基础 / ③ 调研地点 / ④ 归档位置
（多场次策略仅 >1 场时才问；如果这批满 4 题，多场次策略放到第二批）

**Step C · 第二批选项卡**（调研背景 + 调研目的两组多选题）：
两组各 6 个选项 + "自己写"，允许多选。选"自己写"的留到 Step D 一起追问文本。

**Step D · 文本批量追问**（一次性把所有文本项列出来让用户填）：
仍为空的文本项一次问完，**不要单条单条问**。典型清单：
- 客户名称（如首轮未给）
- 调研署名 by-@（用户自己写）
- 调研日期（年月日，如 `2026-04-28`）
- 调研人员名单（我方，顿号分隔）
- 场次排期明细（粘贴原文 / 截图 / 逐条描述都行）
- B 模板额外：调研对象（部门·姓名）+ 调研材料（无则填"无"）
- 背景/目的选了"自己写"的，填具体文本
- 归档选了"其他空间"的，填空间名 + 子夹名

追问时用一段话或一个清单把全部未填项列出来，让用户一次性回复。

**Step E · 解析后复盘确认**（v1.5 强约束，沿用）：
用户提供场次排期、截图、自定义文本后，AI 必须用 Markdown 表格**复盘呈现解析结果**，让用户一眼确认"AI 有没有理解错"，OK 后再进入建档流程。

#### 1.3 智能跳过规则

- 只有 1 场调研 → 不问"多场次策略"
- 用户首轮消息已明确某字段 → 在清单里标 ✅ 并跳过
- 用户上传了截图/粘贴了表格 → 优先解析场次明细，不要再重复问
- **枚举字段**可以给默认推荐值（归档 ① / 地点 ①），但**文本字段（署名/日期/人员/对象/材料）绝不预填、绝不代入**，一律让用户自己写

### Step 2 · 选模板（根据使用基础分叉）

- **A · 用得不好** → 采用《模板 A · 场景挖掘型》，参见 `templates/template_A.mdx`
- **B · 用得好** → 采用《模板 B · 最佳实践型》，参见 `templates/template_B.mdx`

读取对应模板文件后，作为 MDX 骨架填充用户参数。

### Step 3 · 多场次处理

**仅对"正式调研"章节做场次循环**；背景、目的、场次排期总览表始终保持一份（不重复）。

- 只有 1 场 → 单篇文档，正式调研章节只展开 1 场
- 多场次 + 用户选"合成一篇" → 单篇文档，正式调研章节按序号循环展开每一场
- 多场次 + 用户选"每场一篇" → 为每一场分别创建一篇独立文档，文档标题带场次编号（如"01 · 产品部"）

### Step 4 · 参数替换规则

模板中所有 `{{xxx}}` 占位符必须替换为实际参数：

| 占位符 | 含义 |
|--------|------|
| `{{客户名}}` | 客户全称 |
| `{{产品}}` | "腾讯文档" 或 "企微文档"，随用户选择 |
| `{{署名}}` | by-@ 后面的人 |
| `{{调研时间}}` / `{{调研地点}}` / `{{调研人员}}` | 用户提供的基础信息 |
| `{{背景目的}}` | 用户提供或默认模板 |
| `{{场次编号}}` / `{{场次时间}}` / `{{场次部门}}` / `{{场次对象}}` / `{{场次场景}}` | 各场次参数 |

**关键原则**：
- 正式调研部分的**问题文案固定不动**，只替换里面的参数（部门名、用户名、产品名）
- 全文所有"企微文档"/"腾讯文档"的表述要根据 `{{产品}}` 统一
- 所有 emoji 必须保留

### Step 5 · 生成并调用腾讯文档 MCP

1. 对照 `/Users/cathy/.workbuddy/skills/腾讯文档/smartcanvas/mdx_references.md` 逐条自校验 MDX 格式
2. 调用工具创建文档：

```bash
mcporter call "tencent-docs" "create_smartcanvas_by_mdx" --args '{
  "title": "{{客户名}}｜{{产品}}场景调研",
  "mdx": "<完整 MDX 内容>"
}'
```

- 不指定 `parent_id` → 创建到根目录（符合用户要求）
- 多场次每场一篇时，循环调用 N 次

### Step 5.5 · 归档到空间（可选，v1.6 新增）

**何时执行**：仅当用户在 Step B 选择了"归档到 xx 空间"时执行；选"不归档"则跳过本步骤直接 Step 6。

**核心原则**：空间和客户子夹都要**复用优先，不存在再创建**，避免重复建。

#### 5.5.1 定位或创建目标空间

```bash
# 1) 查已有空间列表，看目标空间是否存在
mcporter call "tencent-docs" "query_space_list" --args '{"query_by":1,"order_by":3,"descending":true}'
```

- 在返回的 `spaces` 数组里按 `title` 精确匹配用户指定的空间名
- **找到** → 记下 `space_id`
- **没找到** → 调用 `create_space` 新建：

```bash
mcporter call "tencent-docs" "create_space" --args '{"title":"客户调研｜AI生成","description":"AI 自动生成的客户调研大纲文档归档空间"}'
```

返回的 `space_id` 即为目标空间 ID。

#### 5.5.2 定位或创建客户子文件夹

```bash
# 2) 查空间根节点下的文件夹，看客户名子夹是否存在
mcporter call "tencent-docs" "query_space_node" --args '{"space_id":"<space_id>"}'
```

- 在返回的 `children` 里找 `node_type=wiki_folder` 且 `title={{客户名}}` 的节点
- **找到** → 记下 `node_id` 作为 `target_parent_id`
- **没找到** → 调用 `create_space_node` 新建：

```bash
mcporter call "tencent-docs" "create_space_node" --args '{
  "space_id":"<space_id>",
  "title":"{{客户名}}",
  "node_type":"wiki_folder",
  "wiki_folder_node":{"title":"{{客户名}}"}
}'
```

返回的 `node_info.node_id` 即为 `target_parent_id`。

> ⚠️ **分页注意**：`query_space_node` 每页 20 条。如果 `has_next=true`，需要翻页继续查，避免漏判导致建重复文件夹。

#### 5.5.3 批量移动文档到子夹

对 Step 5 生成的每篇文档 `file_id`，循环调用：

```bash
mcporter call "tencent-docs" "manage.move_file_to_space" --args '{
  "file_id":"<本篇文档 file_id>",
  "space_id":"<space_id>",
  "target_parent_id":"<客户子夹 node_id>"
}'
```

- **必须用 `move_file_to_space`**，不是 `move_file`（后者只能在首页内移动，跨首页→空间会失败）
- 每篇调用一次，返回 `trace_id` 且 `error` 为空即成功
- 批量移动结束后，再调一次 `query_space_node`（带 `parent_id=客户子夹 node_id`）验证 8 篇都在

#### 5.5.4 归档完成后的 URL 结构

- 单篇文档 URL 格式：`https://docs.qq.com/space/D{space_id_hash}?resourceId={file_id}&mode=wiki_mode`
- 归档后 URL 和原 `/aio/{file_id}` 都可访问，**原链接不会失效**，无需替换返回给用户的链接

### Step 6 · 返回结果

向用户返回：
- 每篇文档的**标题**和 **URL**
- 如果执行了 Step 5.5 归档：额外返回**归档位置**（空间名 / 客户子夹）+ **空间入口 URL** + **子夹入口 URL**
- 简短提示："文档已建好，你直接去腾讯文档打开编辑即可，问题文案是固定模板，部门/人员/时间参数我已经替换好了。"
- **⚠️ 必说**：表格默认不满宽，需要**手动拖表格最右侧边框到文档右边**撑开。这是腾讯文档组件限制，已上报。

## 🎨 排版风格（v1.5 最新规则，严格执行）

生成 MDX 时，严格按以下视觉规则：

- **抬头署名** 单独成段，右对齐：`<Paragraph textAlign="right">by-@**xxx**</Paragraph>`
- **保密声明** 用 `<BlockQuote>` 引用块表达，不要塞进 Callout
- **基础信息块**（时间/地点/人员/对象/材料）用 `<Callout blockColor="light_grey" borderColor="grey" icon="🔒">`，内含 BulletedList
- **重要提醒类**（"请提前参阅"等）用 `<Callout blockColor="light_yellow" borderColor="orange" icon="⚠️">`
- **结构纲要 / 核心板块**（调研背景正文、调研目的正文、"调研内容"三板块列表）整段包进 `<Callout blockColor="light_blue" borderColor="blue" icon="💡">` 🆕v1.5
- **说明 / 参考 / 温馨提示类**（"以下问题仅作参考..."、"可提前准备好业务流程图..."等）用 `<BlockQuote>`，**不要用 Callout** 🆕v1.5
- **调研目的段落**：只保留"开头一句 + 对内/对外推广两条 NumberedList"，**不再枚举"场景挖掘/需求收集/最佳实践/流程梳理"等具体子目的**（背景段已表达）🆕v1.5
- **正式调研每个场次的时间+场景**（A 模板）也包进 `<Callout blockColor="light_blue" borderColor="blue" icon="💡">`
- **场次标题** 使用 `#### 01 · 部门 · 对象`（level 4）
- **"问题 1/2/3"** 使用 `##### 问题 N`（level 5），问题文案放紧跟的 `<Paragraph>` 里
- **表格表头** 统一：`<TableCell blockColor="light_grey">` + `<Paragraph textAlign="center">**表头文字**</Paragraph>`
- **场次之间** 不插 `<Divider />`，靠标题层级自然分隔
- **问题之间留白**：每个问题区块结束后（包括末尾表格后），用 `<Paragraph>&nbsp;</Paragraph>` 空段落隔开，避免太紧凑
- **场次排期总览表** 5 列：序号 / 时间 / 部门 / 调研对象 / 智能表场景（**不要加"是否完成"列**）
- 行内强调 **优先用 Markdown `**粗体**`**，不需要 `<Mark bold>`
- **每个 `<TableCell>` 内的文本必须裹 `<Paragraph>`**，否则对齐和加粗会失效

## ⚠️ 已知限制（腾讯文档能力缺失，已上报）

**表格无法程序化满宽**：MDX 规范里 `Table` / `TableRow` / `TableCell` 都**没有 width 属性**，所以通过 API 生成的表格默认按内容宽度渲染，无法自动撑到文档页宽。
- **绕行方案**：文档生成后在腾讯文档里鼠标拖表格最右侧边框到页面右边即可撑到满宽。

**`smartcanvas.read` 无法读取容器内部子 Block**：`Callout` / `BlockQuote` / `ColumnList` 等容器类 Block 内部嵌套的子元素（段落/列表/表格），read 工具不返回，导致 AI 看不到正文。
- **绕行方案**：想确认容器内部是否存在某段内容时，用 `smartcanvas.find` 按关键词检索；read 返回的大小差异不代表用户真的删了内容，**不能只凭 read 结果判断用户改了什么**。
